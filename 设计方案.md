# 钉钉工作汇报统计系统 — 设计方案

## 一、项目背景

从零搭建一个公司内部使用的项目管理系统，核心目标：
- 从钉钉读取员工工作汇报（日报/周报），解析其中包含的项目、工时、工作内容
- 按项目/人员/部门维度汇总统计工时投入
- 基于角色的权限管理（产品经理看负责项目、部门经理看部门成员、高层看全局等）

---

## 二、技术选型

| 层 | 选型 | 理由 |
|---|---|---|
| **后端框架** | Django 5.x + Django REST Framework | 内置 Admin 管理后台、ORM 聚合查询强、权限系统成熟、Celery 调度集成好 |
| **数据库** | PostgreSQL 15+ | JSONB 存原始日志内容、窗口函数优化聚合统计 |
| **缓存/队列** | Redis | 钉钉 Access Token 缓存（2h TTL）、Celery 消息代理 |
| **前端** | Vue 3 + Element Plus + ECharts | 中国企业级 UI 标准套件，交互式仪表盘 |
| **构建** | Vite | 最快的前端开发/构建工具 |
| **部署** | 本地：Docker Compose / 云端：Nginx + Gunicorn | 本地 Docker 一键启动，云端同架构无缝迁移 |

> **为什么选 Django 而非 FastAPI**：内部工具用户量有限（<500），Django 的 Admin 管理后台、ORM 聚合、权限组系统可节省大量开发时间；异步性能优势在此场景不关键。

## 三、数据模型（核心）

```
Department（部门）
  └── User（用户 — 扩展 Django AbstractUser）
        ├── role: admin | executive | dept_manager | product_manager | employee
        ├── dingtalk_user_id / union_id / open_id
        └── department FK

Project（项目）
  ├── product_managers M2M → User
  └── WorkEntry FK ← 工时记录

ReportTemplate（钉钉日志模板，如"日报""周报"）
  └── WorkReport（单条工作日志）
        ├── raw_contents JSONB（原始内容保留，可审计）
        ├── creator FK → User
        └── ReportContent（按字段拆分："今日完成工作"、"明日计划"等）

WorkEntry（解析后的工时记录 — 核心分析单元）
  ├── report FK, employee FK, department FK, project FK
  ├── date, hours, task_description, task_type, status
  └── raw_text（原始文本片段，可追溯）

SyncLog（同步日志 — 记录每次从钉钉拉取数据的结果）
```

### 关键设计决策

- **原始内容 + 解析内容 双存储**：`WorkReport.raw_contents` 保留钉钉返回的原始 JSON，`WorkEntry` 存储解析后的结构化数据，既保证可审计又支持快速聚合查询
- **部门冗余存储**：WorkEntry 同时存 employee 和 department，避免聚合查询时频繁 JOIN
- **项目由管理员创建后匹配**：管理员先在系统中创建项目（名称+别名），同步时从日志文本中匹配已知项目名，匹配不上的条目标记为"未归类"待人工处理

## 四、钉钉集成

### 4.1 SSO 登录流程

```
用户访问 → 前端显示钉钉二维码 → 手机钉钉扫码授权
→ 回调后端 /api/auth/callback/?authCode=xxx
→ 后端用 authCode 换取 userAccessToken
→ 调用 /topapi/v2/user/getuserinfo 获取 userid
→ 创建/更新本地 User 记录 → 签发 JWT → 前端跳转主页
```

### 4.2 日志同步 API

- **接口**: `POST /topapi/report/list`（分页游标，每页最多20条，时间跨度≤180天）
- **Access Token 管理**: 通过 Redis 缓存，TTL=7000秒（2小时减去安全余量），使用分布式锁防止并发刷新

### 4.3 同步调度策略

| 调度 | 频率 | 范围 |
|------|------|------|
| 工作日增量同步 | 每30分钟（9:00-18:00） | 近1天 |
| 每晚深度同步 | 每天 2:00 AM | 近7天 |
| 每周全量对账 | 每周日 3:00 AM | 近30天 |
| 手动触发 | 管理员按需 | 自定义 |

### 4.4 日志内容解析

员工使用钉钉固定模板，实际模板包含 **7 个字段**：

| 字段 | 名称 | 解析价值 | 说明 |
|------|------|:--:|------|
| 字段1 | 今日完成工作 | ⭐ 核心 | 包含项目+工时+内容描述，解析为已完成工时 |
| 字段2 | 今日未完成工作 | ⭐ 有用 | 记录未完成的工作，可按需纳入统计或仅展示 |
| 字段3 | 需协调工作 | 参考 | 阻塞项/需要他人配合的事项，用于项目管理 |
| 字段4 | 明日重点工作计划 | 参考 | 计划性内容，不统计工时但可用于进度跟踪 |
| 字段5 | 备注 | 参考 | 补充说明 |
| 字段6 | 图片 | 跳过 | 附件类，不参与文本解析 |
| 字段7 | 附件 | 跳过 | 附件类，不参与文本解析 |

**解析策略**（因员工填写**格式不统一**，采用多轮渐进式解析）：

1. **字段分类**：通过模板字段名精确识别，仅对"今日完成工作"做深度解析提取工时和项目；"未完成工作"和"需协调工作"留存文本用于进度跟踪
2. **按分隔符拆分条目**：常见的分隔方式包括换行、编号（`1.`、`①`）、符号（`-`、`•`、`；`、`。`），先拆分再逐条处理
3. **逐条目提取**（正则+规则）：
   - **工时**：匹配 `3小时`、`2.5h`、`1.5天`（≤1天按8小时换算）、`半天`（4小时）、`30分钟`（0.5小时）、`3工时` 等常见变体
   - **项目**：与 `Project` 表（含项目名+别名列表）做精确优先/模糊兜底匹配，支持全称、简称、缩写
   - **任务描述**：去除项目名和工时后的剩余文本
4. **置信度评分**：每个 WorkEntry 附带置信度（0-100），规则明确的给高分，模糊匹配的给低分
5. **低置信度人工处理**：低于阈值的条目标记为"**未归类**"，在 UI 提供批量分配和手动修正入口
6. **LLM 兜底**（可选开关）：对格式特别不规范或完全匹配不上的条目，调用大模型提取结构化数据（项目名、工时、任务类型），同时用于智能分类任务类型（开发/测试/会议/文档等）

## 五、权限体系

| 权限 | 管理员 | 高层 | 部门经理 | 产品经理 | 普通员工 |
|------|:--:|:--:|:--:|:--:|:--:|
| 看自己的日志/统计 | ✅ | ✅ | ✅ | ✅ | ✅ |
| 看部门成员日志/统计 | 全部 | 全部 | 本部门 | ❌ | ❌ |
| 看项目统计 | 全部 | 全部 | 本部门项目 | 负责项目 | ❌ |

| 管理用户/角色 | ✅ | ❌ | ❌ | ❌ | ❌ |
| 管理项目 | ✅ | ❌ | ❌ | ❌ | ❌ |
| 触发同步 | ✅ | ✅ | ❌ | ❌ | ❌ |

### 7.1 Dashboard 仪表盘 — 四视角设计

仪表盘 (`/dashboard`) 采用 Element Plus `<el-tabs>` 四选项卡布局，每个选项卡对应一个独立的分析视角：

| 视角 | 布局 | API `?view=` | 数据结构 |
|------|------|-------------|----------|
| **概览** | 单面板：KPI 卡片 + ECharts 图表 | `summary`（默认） | project/employee breakdown + daily trend |
| **员工视角** | 双栏：左侧员工列表 + 右侧详情 | `employee` | Employee → Project → WorkType + "查看日志"
| **项目视角** | 双栏：左侧项目列表 + 右侧详情 | `project` | Project → WorkType → Employee → Entries |
| **部门视角** | 双栏：左侧部门树 + 右侧详情 | `department` | DeptTree → DeptSummary → Employee → Project → Entries |

#### 项目视角详情

双栏布局（240px 项目列表 + 1fr 详情面板）：

1. **项目汇总层**（`ProjectSummary` 组件）：KPI 卡片（总工时 / 参与人数 / 涉及部门）+ 工作类型分布标签条
2. **按工作类型明细表**（`el-table` + 展开行）：每种工作类型一行 → 展开显示参与员工卡片 → 员工卡片内列出具体条目（日期 / 工时 / 任务描述）

后端 `_get_project_view`：条目级查询（不聚合），Python 内存中按 `Project → WorkType → Employee → Entries` 四层分组。

#### 员工视角详情

双栏布局（240px 员工列表 + 1fr 详情面板）：

1. **员工汇总层**（`EmployeeSummary` 组件）：KPI 卡片（总工时 / 项目数 / 工作类型数）+ 跨项目汇总的工作类型分布标签条
2. **按项目明细表**（`el-table` + 展开行）：每个项目一行 → 展开显示该项目下的工作类型及工时
3. **"查看日志"按钮**：位于员工标题右侧，调起 `EmployeeReportDrawer` 查看完整报告

后端 `_get_employee_view`：保持聚合查询（`.values()` + `Sum`/`Count`），仅增加 `employee_username` 字段透传（1 行改动）。前端 `EmployeeSummary` 通过客户端 computed 跨项目汇总工作类型。

#### 部门视角详情

双栏布局（240px 部门树 + 1fr 详情面板）：

1. **部门汇总层**（`DepartmentSummary` 组件）：KPI 卡片（员工数 / 总工时 / 参与项目）+ 项目工时分布（进度条 + 每个项目的工作类型标签）
2. **员工明细表**（`el-table` + 展开行）：每位员工一行 → 展开显示参与项目卡片（项目名/工时 + 工作类型标签 + 条目明细）→ "操作"列提供"查看日志"按钮

"查看日志"功能（`EmployeeReportDrawer`）：右侧滑出 560px 抽屉 → 分页日志列表（调用 `/api/reports/`）→ 点击 → 日志详情（调用 `/api/reports/{id}/`）：KPI 条 + 按项目分组的工时条目 + 可折叠原始日志内容。

#### 设计系统

- **主题**："Precision Instrument" 暗色主题（`--steel: #1B1F2A` / `--brass: #C8A45C` 等）
- **工作类型颜色**：开发 `#C8A45C` / 测试 `#4A90A4` / 会议 `#7D9B76` / 文档 `#8B7AA0` / 设计 `#D4695A` / 其他 `rgba(255,255,255,0.3)`
- **Element Plus 暗色覆盖**：所有含 el-table/el-tree 的组件必须使用 non-scoped `<style>` 块覆盖 CSS 变量（透明背景 + 禁止 hover/focus 效果）

## 十、关键风险 & 缓解

| 风险 | 缓解措施 |
|------|----------|
| 日志文本解析准确率低 | 置信度评分 + 低置信度条目标记人工复核；可选 LLM 兜底 |
| 钉钉 oapi 接口未来变更 | API 客户端抽象为接口层，切换时仅修改 client.py |
| 用户尚无钉钉开发者凭证 | 内置 Demo 模式：本地账号登录 + JSON fixture 数据，可先行开发 |
| authCode 有效期极短（~5秒） | 后端收到回调后立即换取 token，不做任何其他操作 |

## 十一、部署方案（本地测试 → 云端上线）

### 11.1 本地开发环境（Phase 1 目标）

开发者的 Windows 电脑上通过 Docker Compose 一键启动全部服务：

```
┌──────────────────────────────────────────┐
│                Docker Compose             │
│  ┌─────────┐  ┌──────────┐  ┌────────┐  │
│  │ Django  │  │  Celery  │  │  Vue   │  │
│  │ :8000   │  │  Worker  │  │ :5173  │  │
│  └────┬────┘  └────┬─────┘  └────────┘  │
│       │            │                     │
│  ┌────┴────────────┴──────┐              │
│  │     PostgreSQL:5432     │              │
│  └────────────────────────┘              │
│  ┌────────────────────────┐              │
│  │      Redis:6379         │              │
│  └────────────────────────┘              │
└──────────────────────────────────────────┘
```

**一键启动命令**：
```bash
# 前提：已安装 Docker Desktop + Git
git clone <仓库地址> Hisky
cd Hisky
cp .env.example .env          # 填入钉钉密钥（可先用 Demo 模式跳过）
docker-compose up -d           # 启动全部服务
# 浏览器打开 http://localhost:5173
```

**Demo 模式**（无需钉钉凭证即可开发测试）：
```bash
# .env 中设置
DINGTALK_DEMO_MODE=True       # 跳过钉钉 SSO，使用本地账号登录
```
- Demo 模式下使用 Django 内置登录（admin/admin123）
- 同步功能读取 `fixtures/demo_reports.json` 模拟数据
- 所有功能（仪表盘、解析、权限）均可正常测试

### 11.3 CI/CD 流程（云端阶段启用）

```
Git Push (main 分支)
  → GitHub Actions / GitLab CI
    → 后端：pytest 测试
    → 前端：npm run build 构建
    → 构建 Docker 镜像 → 推送镜像仓库
    → SSH 云服务器 → docker-compose up -d
    → 数据库 migrate + 健康检查
```
